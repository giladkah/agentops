How the Claude Code team works
01 Open 3–5 terminals, each with its own worktree and independent Claude session
02 Each session tackles the same problem independently — no shared context, no bias
03 One Claude reviews the plan as a staff engineer before implementation starts
04 Compare what comes back. The answer that holds up across runs is the one you trust.
What Ensemble does
→ Spins up N isolated worktrees automatically — no terminal juggling, no manual setup
→ Runs each session fully independently — 2, 3, 4 runs, as many as you want
→ Adds a specialized review layer to each run — security, architecture, senior eng
→ Surfaces only what survives. Human approval before anything merges. Every time.

// how it works

How Ensemble works

The first pass is easy. The scrutiny is the product.

01
Signal → triage
A task, issue, or error enters. The codebase is analyzed, affected files mapped, context built.
02
Parallel worktrees
N isolated worktrees spin up. Each gets its own independent Claude session — no shared context, no bias toward the first answer.
03
Review + compare
Each run passes through Senior, Security, and Architecture review. Results compared. Drift flagged. Weak runs excluded.
04
Human approval → PR
Your team reviews what survived. One PR, backed by convergence. Nothing merges without your sign-off.
Senior Engineer
Implementation quality, correctness, patterns — does this change make sense in context?
Security
Auth issues, secrets exposure, injection risks, trust-boundary problems — caught before merge.
Architecture
System boundaries, maintainability, design tradeoffs — does this fit the shape of the codebase?
